# CAMSYS v1.5.4 — патч (нарастающий, поверх v1.4.3)

## Что нового с v1.5.3 — три бага в drill'е реперов

Все три существовали и до v1.5.x, но проявились сейчас, потому что
POSITION-вариант привлёк внимание к содержимому drill-цикла.

### Баг 1: эмиттер сверлил только первые 2 репера

`_emit_fiducial_drill` в `post/mtx_anderson.py` жёстко брал `points[0]` и
`points[1]`, игнорируя `points[2:]`. Для заказов с 3 реперами (пример:
121561 = FID5+FID6+FID7) третий репер молча пропадал.

Исправлено: цикл по всем точкам с корректными тегами `;210,1..4` для
первого прокола и `;211,2 → ;211,3 → ;211,1` для каждого следующего.

### Баг 2: обычный экспорт сверлил реперы других заказов

В `_generate_rough_all` (package_export.py) drill-op пересобирается из
`prj.fiducials` с фильтром по `_fiducial_excluded_ids`. А
`_fiducial_excluded_ids` учитывал только галку `excluded` (снятую
оператором в UI) и НЕ учитывал `stitch_filtered_out`. Из-за этого при
экспорте 121561 сверлились ещё и реперы 121554/121555.

Исправлено: `_fiducial_excluded_ids` теперь включает и
`stitch_filtered_out`. Обычный экспорт заказа сверлит только его
собственные реперы.

### Баг 3: точки drill дублировались

`_generate_rough_all` append'ит агрегированную drill-op (все точки в
`drill_points`) поверх per-fiducial операций (каждая с одной точкой).
Эмиттер собирал точки из ВСЕХ FIDUCIAL_DRILL-операций в шейпе — и
получал каждую точку дважды. Пока эмиттер обрабатывал только 2 точки,
это не было видно; после фикса бага 1 — стало.

Исправлено: дедупликация точек в эмиттере с округлением до 4 знаков.

- Версия `1.4.3 → 1.5.4`.
- Zoom колесом с центром под курсором (`ui/viewer_2d.py`).
- Комментарии `;40,20`/`;50,21`/`;60,21`/`;40,10` на теле контура.
- POSITION-вариант: `CamSession.export_package_position(...)`,
  `core/position.py`, автоматический вызов из UI при нажатии «Экспорт».

- Фикс режима доработки `_dop.anc`: `_total_blade_count` считается в
  пределах scope текущего экспорта (v1.5.3).
